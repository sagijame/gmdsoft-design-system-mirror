사용성

업무형 화면을 오래 쓰는 사람이 빠르게 읽고, 정확히 판단하고, 언어가 바뀌어도 같은 의미를 이해하도록 한다.

1

고밀도 업무 정확성이 우선이다

데스크톱과 웹의 업무형 화면은 많은 정보를 오래 보고 판단하는 환경이다. 장식보다 스캔, 비교, 반복 작업의 정확성을 우선한다.

2

토큰이 단일 출처다

색·크기·간격은 전부 tokens.css의 --gmd-* 변수에서 온다. 화면에 임의 HEX나 px를 직접 박지 않는다.

3

4px 그리드와 제한된 타입 스케일을 쓴다

간격은 4px 리듬을 기본으로 하고, 글자는 허용된 크기 단계 안에서만 쓴다. 조밀하지만 흔들리지 않는 화면을 만든다.

5

컴포넌트를 먼저 쓴다

Button, Input, Card, Modal 같은 Design Kit 컴포넌트 구조와 상태를 우선 사용한다. 새 패턴은 기존 컴포넌트로 설명되지 않을 때만 만든다.

9

Figma Design Kit과 정합해야 한다

문서와 제품 UI는 Figma Design Kit의 컴포넌트, 토큰, 상태와 맞아야 한다. 어긋나는 값은 새 규칙이 아니라 검토 대상이다.

시각성

차분한 업무형 인상 안에서 색, 아이콘, 깊이를 필요한 만큼만 써서 정보 위계와 상태를 드러낸다.

4

blue-gray로 차분하게 깐다

바탕은 cool 톤 blue-gray다. 오래 봐도 피로하지 않고 신뢰를 주기 위해서다. 강조색은 꼭 필요한 곳에만 쓴다.

7

아이콘은 기능을 번역한다

아이콘은 장식이 아니라 텍스트 라벨과 함께 쓰는 24px UI 기호다. 메타포가 기능과 맞아야 하고, 단순한 모노라인 구조를 유지한다.

8

깊이는 명도로 표현한다

요소의 elevation은 무채색 알파 그림자 단계로만 나타낸다. 색이 섞인 그림자는 정보 색을 오염시켜 판단을 흐린다.

간격 (4px 그리드)

요소 사이 여백과 패딩을 4px 단위로 끊어 쓴다. 모든 간격이 4의 배수라 화면 리듬이 일정하고, 디자이너와 개발자가 같은 값을 공유한다. 임의 여백 대신 이 토큰을 먼저 본다.

모든 margin · padding · gap은 4의 배수. 비 4배수는 경고로 표시된다.

Radius

모서리를 얼마나 깎을지 정한다 — none 0·sm 4·md 8·lg 12·xl 16·circle. 입력·버튼은 작게, 카드·모달처럼 큰 면일수록 크게 줘 크기에 맞는 둥글기를 일관되게 맞춘다.

28 컴포넌트

토큰으로 조립된 UI 컴포넌트 28종. 각 항목은 Figma v2.12와 MD-UX 프로젝트의 Docs / * 문서를 미러링한다.

Button

눌러서 작업을 실행하는 컨트롤. 한 화면의 주 동작은 채움형 Primary 하나만 두고, 덜 중요한 동작은 외곽선·텍스트로 위계를 낮춘다. Figma 위계명은 Primary·Secondary·Neutral·Text Link 4단이다.

위계 (Hierarchy)

Figma 위계는 Primary / Secondary / Neutral / Text Link 4단. 클래스명 btn-tertiary=Neutral, btn-text=Text Link로 매핑한다(사용처 보존). btn-danger는 위계가 아닌 semantic destructive로, warning/error Dialog의 확인 버튼에만 쓴다.

상태

각 위계는 default / hover / pressed / disabled 4단을 가진다. Primary hover --gmd-primary-hover, pressed --gmd-primary-pressed, disabled --gmd-primary-disabled.

아이콘

leading 아이콘은 동작 보강, trailing 화살표는 다음 단계 이동을 뜻한다. 아이콘은 라벨과 gap 4px.

크기 (Size)

Figma 크기 6단 — btn-xm 24 · btn-sm 32 · btn-md 36(기본) · btn-lg 40 · btn-xl 48 · btn-2lg 52px. 클래스 없이 btn만 쓰면 md(36)다. 조밀 영역은 xm·sm, 강조 영역은 lg·xl·2lg. Text 버튼은 Download/Downloading/Failed 같은 상태 전환 라벨에도 쓰며, hover·pressed 시 --gmd-primary-subtle pill 배경이 생긴다.

Anatomy

  1. 1Container버튼 외곽. 높이·padding·radius·상태 배경을 담당한다.
  2. 2Label동작을 설명하는 텍스트. Bold 14px, 위계별 텍스트 토큰을 사용한다.
  3. 3Icon선택 구성 요소. leading은 동작 의미, trailing은 이동·다음 단계 의미를 보강한다.

    다국어 잘림 정책

    컴포넌트를 실제 화면에 조립할 때 따르는 운영 규칙. 글로벌 배포 언어에서 텍스트가 컨트롤보다 길어질 때의 처리 기준을 다룬다.

    다국어 잘림 (Truncation)

    같은 의미라도 언어에 따라 길이가 다르다. 독일어·러시아어 같은 언어에서는 짧은 단어가 2~3배까지 길어진다. 모든 잘림을 막을 수는 없으므로 기준은 하나다 — 잘리더라도 전체 내용을 확인할 방법이 항상 있어야 한다.

    불변 조건 (Invariants)

    1

    클리핑 금지

    글자가 중간에 뚝 끊기면 안 된다. 잘림은 항상 말줄임표(…)로 표시한다.

    2

    전문 확인 수단

    잘린 텍스트는 마우스 오버 툴팁으로 전체 내용을 확인할 수 있어야 한다.

    3

    주요 액션은 잘리지 않는다

    실행·저장·삭제 같은 주요 버튼 라벨은 어떤 지원 해상도에서도 온전히 보여야 한다.

    기본 화면(1280×720)과 패널 기본 크기에서는 잘림 없이 표시한다. 사용자가 그보다 줄이면 위 세 조건만 보장한다. 잘린 글자는 최소 4자가 남아야 하고, 페이지 제목·오류 메시지·알림은 자르지 않는다.

    폭 규칙 (Width) Design Kit 반영 검토 대상

    컨트롤이 가질 수 있는 가장 좁은 폭과 가장 넓은 폭. 최소 폭보다 좁아지면 내용을 알아볼 수 없고, 최대 폭에 닿으면 폭을 더 늘리는 대신 더 짧은 번역으로 바꾼다. 값은 tokens.css의 --gmd-width-* 토큰.

    최대 폭에 닿은 버튼은 글자를 자르지 않고 더 짧은 번역으로 바꾼다. 폭을 늘릴 때는 라벨이 긴 그 버튼만 늘리고, 옆 버튼을 따라 넓히지 않는다.

    처리 수단 우선순위

    1. 1레이아웃으로 해결폭을 늘리거나 줄을 바꾼다. 텍스트를 건드리지 않는 가장 안전한 방법.
    2. 2다국어 리소스 축약더 짧은 표현으로 바꾼다. 용어는 Confluence 공식 표기를 따른다.
    3. 3말줄임 + 툴팁위 두 가지가 어려울 때 쓰는 마지막 표시 방법. 주요 버튼에는 쓰지 않는다.
    4. 4컨트롤 구조 변경비용이 가장 큰 수단. 효과를 따로 판단해 결정한다.

    근거

    • W3C 텍스트 팽창 데이터 — 10자 이하 문자열은 200~300%, 70자 이상도 130%까지 길어진다
    • IBM Carbon — 잘린 항목은 항상 툴팁 제공, 최소 4자 유지, 제목·오류 메시지는 잘림 금지
